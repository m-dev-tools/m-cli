# m-cli de novo rebuild — framework selection & proposal

A market survey of production-grade open-source CLI frameworks and a
concrete recommendation for rebuilding `m-cli` from scratch on a
schema-first, aesthetic-first foundation. Don't roll our own — pick the
best off-the-shelf stack and adopt it whole.

Companion to (and supersedes the "build it yourself" arc of):

- [`cli-output-audit-2026-05-13.md`](cli-output-audit-2026-05-13.md)
- [`audit-m-engine-2026-05-13.md`](audit-m-engine-2026-05-13.md)
- [`audit-remediation-plan0-2026-05-13.md`](audit-remediation-plan0-2026-05-13.md)
- [`cli-schema-architecture-2026-05-13.md`](cli-schema-architecture-2026-05-13.md)

The remediation plan and schema-architecture doc are pragmatic: how
to fix what we have. **This document asks the opposite question**:
given the audits caught ~30 inconsistencies across 14 subcommands, and
given m-cli's surface is small enough that a clean rewrite is
realistic, what would a 2026-quality CLI built de novo look like, and
which off-the-shelf framework would carry the most weight?

## 1. Premise & selection criteria

The audits surfaced that m-cli today is in a state where a comprehensive
rehab is on par, in effort, with a de novo rebuild. Once we accept that,
the question becomes: *which existing, well-maintained, broadly-adopted
framework should we adopt?* The non-negotiables:

1. **Schema-first.** Command surface, args, flags, outputs, exit codes
   declared in a manifest the framework reads. Not "argparse with
   metadata sprinkled in."
2. **Used by serious companies, large user bases.** Stripe, GitHub,
   Heroku, Salesforce, Cloudflare, Vercel, HashiCorp, Charm, Datadog,
   etc. Survivorship bias is a feature here — they have the failure
   modes already debugged.
3. **Aesthetic facilities out of the box.** Colour, tables, progress,
   markdown, boxes, spinners, syntax highlighting — without rolling
   our own ANSI codes.
4. **Portable distribution.** Ideally single static binary. Falling
   back to "single `pip install` or `npm install`" is acceptable.
5. **Plugin model.** Out-of-tree subcommands ship without forking the
   core.
6. **Active maintenance.** Last release ≤ 6 months ago, multiple
   contributors, healthy issue triage.
7. **Don't roll our own.** The deal is to inherit, not to invent.

Soft preferences:

- Manifest is JSON or YAML (machine-readable, lintable, version-controlled).
- Output is TTY-aware by default (pretty when interactive, structured
  when piped).
- Help, completion, and reference docs are autogenerated from the
  manifest.
- Errors are styled and actionable (Rust-compiler-quality is the bar).

Two clusters jump out:

- **Go + Cobra** is the dominant stack for "modern systems CLIs"
  (Stripe, gh, fly, Tailscale, Doppler, kubectl, Helm, Hugo, Docker).
- **TypeScript + Oclif or Ink** is the dominant stack for "developer
  tooling CLIs" (Heroku, Salesforce, Vercel, Wrangler, Prisma).

Rust is rising fast (cargo's success, ripgrep's reach) but has fewer
"big company" CLIs because younger teams tend to pick Go.

Python (Click/Typer/Rich) is excellent for tooling that's tightly
coupled to a Python codebase. It loses on startup speed and portability
versus Go/Rust.

## 3. Aesthetic & cognitive design patterns from the field

What separates a CLI that feels modern from one that feels 1995?

### 3.1 The 14 patterns of 2024-26 CLIs

1. **Verb-noun grammar.** `gh pr list`, `stripe customers create`,
   `kubectl get pods`. Not `gh list-prs`, not `stripe create-customer`.
2. **Progressive disclosure.** Bare invocation prints a friendly
   overview; `--help` prints full detail; `--help --verbose` prints
   exhaustive reference. Three tiers.
3. **TTY-awareness.** When stdout is a terminal: colour, tables,
   progress bars, em-dashes. When stdout is piped: plain ASCII, JSON
   if `--format json` is implied. **Detect, don't ask.**
4. **Inline examples in every `--help`.** GitHub CLI's gold standard:
   every command's `--help` shows 2-4 real example invocations under
   an `EXAMPLES` heading.
5. **Coloured but never gaudy.** Three colour roles only: green
   (success), yellow (warning), red (error). Dim grey for auxiliary.
   Bold for emphasis. **No rainbow.**
6. **Errors as compiler diagnostics.** Rust compiler set the bar:
   error line numbers, source excerpts, fix hints. CLIs adopting this:
   wrangler, prisma, deno.
7. **Semantic spinners and progress.** Spinners only for indeterminate
   work > 500ms. Progress bars with ETA only when meaningful.
   *Don't* spin during a 50ms operation.
8. **Glyphs reserved for status.** `✓` (success), `✗` (failure),
   `⚠` (warning), `→` (transition), `…` (truncated), `⠋` (spinner).
   Don't use them as decoration.
9. **Tables that respect terminal width.** Auto-fit columns, truncate
   long values with `…`, hide secondary columns on narrow terminals.
10. **First-run onboarding.** Detect `~/.config/<cli>/` absent → print
    a friendly "looks like this is your first time" with 2-3 commands
    to try. **Once**, never again.
11. **No prompts for non-interactive shells.** When stdin isn't a TTY,
    fail loud rather than hang waiting for input.
12. **Structured exit codes.** 0 success, 1 domain error, 2 usage
    error, 130 SIGINT, 137 SIGKILL. Documented per command in `--help`.
13. **Markdown in `--help` when terminal allows.** Stripe and gh render
    headings and code blocks in `--help` when stdout is a TTY.
14. **Quiet by default, verbose on demand.** `-v` adds detail; `-q`
    removes everything except errors. **Default sits between.**

### 3.2 Anti-patterns to avoid

- **Animated logos** on startup. Wastes time, breaks pipes.
- **Bell characters** (`\a`). Annoying, accessibility hostile.
- **Emoji as load-bearing status.** Some terminals render them
  half-width and break alignment. Use Unicode glyphs (`✓`) not emoji
  (`✅`) for status; emoji are fine for one-off accents.
- **Forced screen-clears.** `clear` on startup destroys scrollback.
- **Unicode in error messages without a fallback.** Some CI logs are
  ASCII-only.
- **Telemetry without opt-out.** Honour `DO_NOT_TRACK=1`.
- **Help text longer than `$LINES`.** Paginate or shorten.

### 3.3 Cognitive design — the seven heuristics

From the human-factors literature on CLI design (Nielsen's heuristics
applied to terminals, plus more specific work from the Microsoft Cmd
team's `PSReadLine` and the Charm/Heroku UX writeups):

1. **Visibility of system status.** Long operations show progress;
   silent commands print at least "done" or "no changes."
2. **Match between system and real world.** Verbs match domain
   vocabulary. `m engine start` not `m engine ignite`.
3. **User control and freedom.** Destructive operations require
   `--confirm`; signals (`Ctrl-C`) leave clean state.
4. **Consistency and standards.** Same flag names mean the same thing
   across commands. `--format json` works everywhere or nowhere.
5. **Error prevention.** Dry-run mode by default for risky operations
   (`m fmt` had this with `--check`; `m ci init` got it later via `--write`).
6. **Recognition rather than recall.** `--help` lists examples; user
   doesn't memorize flag order.
7. **Aesthetic and minimalist design.** Don't print what isn't useful.
   Two-column key-value beats five-column tables when only two
   columns matter.

These map directly onto the audits' findings: today's m-cli violates
#4 (consistency), #5 (`m fmt --check` exits 0 on missing file), #7
(verbose JSON when text would do).

## 4. Market survey — CLI frameworks by ecosystem

Star counts and download stats as of early 2026, rounded. The point
isn't precision; it's relative gravity.

### 4.1 Go ecosystem

| Framework / Library    | Stars  | Used by                                              | Schema-first? | Notes                                                  |
| ---------------------- | -----: | ---------------------------------------------------- | :-----------: | ------------------------------------------------------ |
| **Cobra**              | ~37k   | kubectl, Hugo, gh, Docker, Helm, etcd, Stripe CLI, fly, Tailscale, Doppler | partial (JSON dump available) | The de facto Go CLI framework. POSIX flags, completion gen, command tree. |
| **urfave/cli**         | ~22k   | (Many smaller CLIs)                                  | partial       | Older alternative to Cobra. Simpler API.               |
| **kingpin** (alecthomas) | ~3.6k | Some K8s tooling                                    | partial       | Strong type-driven flag parsing.                       |
| **mitchellh/cli**      | ~1.5k  | Terraform (originally), Consul, Vault                | no            | Internal HashiCorp framework.                          |
| **spf13/pflag**        | ~2k    | (Cobra dependency)                                   | n/a           | POSIX-compliant flag parsing.                          |

**Cobra dominates.** If you write a Go CLI in 2026 and want users to
recognise the conventions, you use Cobra. It has:

- Hierarchical commands via `Command` struct trees.
- Auto-generated bash/zsh/fish/powershell completion.
- `cobra-cli` scaffolds new commands.
- JSON command-tree dump via custom traversal (~30 lines of glue) —
  schema-export is straightforward but not built in.

### 4.2 TypeScript / Node ecosystem

| Framework            | Stars | Used by                                          | Schema-first? | Notes |
| -------------------- | ----: | ------------------------------------------------ | :-----------: | ----- |
| **Oclif**            | ~9k   | Heroku, Salesforce (`sf`), Twilio, Fastly, npm CLI generator | **yes** | `oclif.manifest.json` is autogenerated; plugins ship manifest fragments |
| **Commander.js**     | ~26k  | (Many — vue-cli, nodemon, lerna)                 | no            | The "argparse" of Node; ubiquitous but not schema-driven |
| **yargs**            | ~10k  | nyc, husky, jest                                 | partial       | More features than Commander; less schema       |
| **Citty** (UnJS)     | ~1.5k | Nuxt CLI                                         | partial       | Type-driven, newer entrant                       |
| **clipanion**        | ~1.3k | Yarn Berry                                       | partial       | Class-based; strong types                        |
| **stricli** (Bloomberg) | ~1k | Bloomberg's internal CLIs                       | **yes**       | Schema-first, multi-platform, ESM-native (2024+) |

**Oclif** is the schema-first standout. Its `manifest.json` is the
literal artifact the m-cli audits begged for. Plugin architecture is
first-class. Used by both Heroku (origin) and Salesforce's `sf`
(~3,000+ commands).

### 4.3 Rust ecosystem

| Framework / Library  | Stars | Used by                                          | Schema-first? | Notes                                          |
| -------------------- | ----: | ------------------------------------------------ | :-----------: | ---------------------------------------------- |
| **clap**             | ~14k  | ripgrep, fd, bat, exa/eza, cargo (partial), tokei | partial       | The de facto Rust CLI lib. Derive macros = "type is the schema." |
| **argh** (Google)    | ~2k   | (Internal Google tooling)                        | partial       | Lighter-weight clap alternative                |
| **structopt**        | (folded into clap v3) | n/a                                | partial       | Predecessor to clap's derive API               |
| **bpaf**             | ~1k   | (Niche)                                          | partial       | Applicative-style; powerful but unusual        |

**clap with derive** is the gold standard for Rust. The schema is
encoded in Rust structs (typed); flag parsing is generated. Export to
JSON manifest is possible via `clap_complete::dynamic` but not
turnkey.

### 4.4 Python ecosystem

| Framework              | Stars | Used by                                          | Schema-first? | Notes                                          |
| ---------------------- | ----: | ------------------------------------------------ | :-----------: | ---------------------------------------------- |
| **Click**              | ~16k  | pip, Black, Flask, Poetry (cleo wraps), AWS CLI v2 | partial     | Decorator-based; well-maintained               |
| **Typer**              | ~16k  | FastAPI (sibling), various Anthropic tools       | partial       | Built on Click; type-hint-driven               |
| **Fire** (Google)      | ~27k  | Internal Google tooling                          | no            | Auto-generates CLI from any object             |
| **docopt**             | ~8k   | (Many smaller tools)                             | yes (text)    | "Schema *is* the help text"; mostly unmaintained |
| **cleo** (Poetry)      | ~1.4k | Poetry, Tortoise ORM                             | partial       | Older; Symfony-Console inspired                |
| **argparse** (stdlib)  | n/a   | (Everything else)                                | no            | What m-cli uses today                          |

**Typer + Rich** is the modern Python combo. Type hints drive flag
parsing; Rich renders gorgeous output. Used by many recent CLIs
including FastAPI's own (uvicorn isn't Typer but FastAPI's `fastapi
dev` is).

Schema-first in the Oclif sense doesn't really exist in Python yet.
The closest is Typer's introspection helpers, but they're code-level,
not external-manifest.

### 4.5 Other

- **Deno's flag parser** (`@std/cli`) — type-driven, modern, but
  Deno-only.
- **Bun's `command-line-args`** — too new.
- **Crystal's OptionParser** — niche.
- **Elixir's OptionParser** — niche.
- **PowerShell** — built-in Verb-Noun, approved-verbs list. Not really
  cross-platform-portable for a CLI like m.

## 5. Aesthetic / TUI libraries

The framework is half the story. The other half is the rendering
library — how good can the output *look* with reasonable effort?

### 5.1 Go — the Charm ecosystem

**Charm.sh** has redefined Go CLI aesthetics since ~2021. Stats are
combined across the family:

| Library          | Stars | Purpose                                                                  |
| ---------------- | ----: | ------------------------------------------------------------------------ |
| **Lip Gloss**    | ~9k   | Declarative styling — borders, padding, colours, layouts                 |
| **Bubble Tea**   | ~30k  | Elm-architecture TUI framework (interactive apps)                        |
| **Bubbles**      | ~6k   | Reusable TUI components (text inputs, lists, tables, spinners, progress) |
| **Glamour**      | ~3.4k | Markdown rendering in the terminal                                        |
| **Huh**          | ~6k   | Forms / prompts (replaces survey)                                        |
| **Glow**         | ~18k  | Markdown reader (showcase)                                               |
| **Gum**          | ~22k  | Shell-scripting-friendly TUI components                                  |
| **VHS**          | ~16k  | Record terminal sessions as GIF/MP4 (for docs)                           |

Charm's combined polish is the high bar of CLI aesthetics in 2024-26.
Build with Cobra + Lip Gloss + Glamour and you get a CLI that *looks*

### 5.2 Python — the Textualize ecosystem

| Library        | Stars | Purpose                                                                    |
| -------------- | ----: | -------------------------------------------------------------------------- |

Rust's TUI ecosystem is a notch behind Charm and Ink in turnkey

### 5.5 Cross-cutting: what "modern" requires

Whichever stack we pick, the rendering layer should give us:

- **Borders & boxes** (Lip Gloss / Rich / Ink / ratatui all do this).
- **Tables** (responsive width, auto-fit columns).
- **Progress bars + spinners** (with ETA, multi-line).
- **Markdown** (for `--help` bodies and inline docs).
- **Syntax highlighting** (for M code snippets in lint output, fmt
  diffs, doc examples).
- **Diff rendering** (unified diff, side-by-side; m-cli's `m fmt --diff`).
- **Trees** (for `m capabilities`, `m engine status`, etc.).
- **TTY-detection helpers** (so plain stdout is used when piped).

The Charm and Rich ecosystems both cover all of this. Ink covers it
slightly less ergonomically (React idioms add ceremony). ratatui is
the lowest-level of the three.

## 6. Finalist stacks compared

After cuts, four real candidates:

### 6.1 **Finalist A — Go + Cobra + Charm**

```
Language:          Go 1.22+
CLI framework:     spf13/cobra (~37k stars)
Aesthetic stack:   charmbracelet/lipgloss (~9k) + bubbletea (~30k)
                   + glamour (~3.4k) + huh (~6k) + bubbles (~6k)
Config:            spf13/viper (sibling of cobra)
Schema export:     custom JSON dumper (~50 LOC) walking cobra tree
Plugin model:      Cobra subcommand registration; out-of-tree via
                   `$PATH` discovery (`m-foo` → `m foo`)
Distribution:      single static binary (`go build`)
Startup time:      ~3-10 ms cold
```

**Strengths**
- Single static binary. `curl https://… | tar` and m is on `$PATH`.
- Cobra is what every modern systems CLI uses. Cognitive familiarity
  for users is highest.
- Charm gives best-in-class aesthetics with low ceremony.
- Sub-10ms startup vs current Python ~150ms.
- Cross-platform out of the box (Linux/macOS/Windows).
- M-tree-sitter has Go bindings (tree-sitter-go is first-class).
- Cobra completion gen is mature: bash/zsh/fish/powershell.

**Weaknesses**
- **m-cli's Python engines** (lint rules in `_modern.py`, fmt rules in
  `fmt/rules.py`, the `_keywords.py` m-standard loader, the tree-sitter
  AST walkers) would need to be **rewritten in Go** or kept as **Python
  workers** (subprocess + JSON-RPC). Neither is free; both are
  realistic.
- Cobra's schema export is good but not as elegant as Oclif's
  `oclif.manifest.json`. Glue code required.
- The wider Python tooling (`uv`, `ruff`, `mypy`, `pytest`) becomes
  irrelevant for the CLI itself.

### 6.2 **Finalist B — TypeScript + Oclif + Ink**

```
Language:          TypeScript 5+
CLI framework:     oclif (~9k stars, Heroku/Salesforce)
Aesthetic stack:   ink (~28k) + ink-spinner + ink-table + chalk
Markdown rendering: marked-terminal / marked + Ink
Config:            cosmiconfig
Schema export:     `oclif manifest` → `oclif.manifest.json` (built in)
Plugin model:      first-class via Oclif plugins (npm packages)
Distribution:      `npm i -g`, `pkg`-packed binary, or Bun single-file
                   executable
Startup time:      ~150-250 ms cold (Node startup overhead)
```

**Strengths**
- **The most schema-first option on the table.** `oclif.manifest.json`
  is THE artifact m-cli's audits asked for — generated automatically,
  describes every command, flag, and arg across the CLI and all
  installed plugins.
- Plugin model is the most mature in the industry (Heroku/Salesforce).
- Ink's React model is ergonomic for complex output (status dashboards,
  progress trees).
- Lots of "company-grade" prior art to crib from (Vercel, Wrangler,
  Prisma, sf).
- TypeScript types double as runtime validation via zod / valibot.

**Weaknesses**
- Node startup is ~150-250 ms. m-cli today is ~150 ms. **No
  improvement** unless Bun is adopted (10ms startup, but newer
  ecosystem).
- m-cli's Python engines still need to be either rewritten in TS or
  invoked as subprocess workers (same as Go option).
- tree-sitter has JS bindings but they're slower than Go/Rust ones.
- "Install Node to run an M tool" feels heavier than a static binary.

### 6.3 **Finalist C — Rust + clap + ratatui**

```
Language:          Rust 2024 edition
CLI framework:     clap v4 (~14k stars) with derive macros
Aesthetic stack:   ratatui (~10k) + indicatif (~4.6k) + owo-colors
                   + comfy-table
Markdown:          termimad / mdcat (subprocess) or pulldown-cmark
Config:            figment / config
Schema export:     clap_complete + custom dumper
Plugin model:      Rust ABI is unstable; plugins via subprocess discovery
Distribution:      single static binary, smallest of any option (~5 MB)
Startup time:      ~1-3 ms cold
```

**Strengths**
- Fastest startup of any candidate (sub-5 ms).
- Smallest binary. Best portability.
- clap's derive macros are the closest thing in any ecosystem to
  "types are the schema."
- Rust's tree-sitter bindings are the **canonical** ones — every other
  language's tree-sitter is a re-binding.
- Strong type system catches bugs at compile time.
- Strong ecosystem alignment with where M-tooling could go (m-language
  could grow native-compiled tools).

**Weaknesses**
- **Largest rewrite cost.** Rust has the steepest learning curve and
  the slowest "make it work" loop.
- ratatui is great but not as turnkey as Charm.
- Plugin model is the weakest of any finalist.
- Smaller community vs Go/TS for CLI specifically. Most popular Rust
  CLIs (ripgrep, fd, bat) are minimal "one job" tools, not large
  multi-verb suites.

### 6.4 **Finalist D — Python + Typer + Rich (stay)**

```
Language:          Python 3.12+
CLI framework:     typer (~16k stars) — built on click
Aesthetic stack:   rich (~50k) + textual (~26k for any TUI bits)
Schema export:     click-introspection helpers + custom dumper
Plugin model:      Python entry points (today's model)
Distribution:      `uv tool install m-cli` or `pipx install m-cli`
Startup time:      ~100-200 ms (Python interpreter)
```

**Strengths**
- **Smallest migration cost by far.** The Python engines stay where
  they are. Typer is a near-drop-in replacement for argparse with
  better ergonomics.
- Rich is the single best terminal-rendering library in any language.
  Tables, progress, syntax highlighting, JSON-pretty-print, tracebacks,
  markdown — all turnkey.
- Textual exists for any future TUI ambitions.
- Plays nicely with the existing pyproject.toml/uv ecosystem.

**Weaknesses**
- **Not truly schema-first.** Typer derives flags from type hints,
  which is closer to "schema" than argparse but still code-as-spec.
  Manifest must be exported, not authored.
- Python startup overhead. ~150 ms cold; with Rich loaded ~250 ms.
- Single static binary is hard. `uv tool install` is the modern fix
  but still requires Python.
- No first-class plugin manifest like Oclif's.

### 6.5 Side-by-side scorecard

## 7. Recommendation

Two recommendations, depending on appetite.

### 7.1 The realistic recommendation — Finalist D (Typer + Rich), reimagined

If the *only* goal is fixing what the audits found, **don't rewrite
the language**. Adopt:

- **Typer** to replace argparse (16k stars, FastAPI sibling, type-hint
  driven, drop-in compatible with argparse mental model).
- **Rich** for all output (50k stars, *the* Python terminal library).
- **Pydantic / msgspec** for output-schema validation (manifests authored
  as Pydantic models; JSON Schema autogenerated).
- **The remediation plan's Plan-0** for everything else.

This delivers ~80% of the aesthetic and schema gains at ~20% of the
rewrite cost. The Python engines (lint, fmt, tree-sitter, m-standard)
keep working unchanged.

**This is what I'd actually do.** It's the responsible answer for a
single-developer hobbyist project where the audits found process
problems, not language problems.

### 7.2 The aspirational recommendation — Finalist A (Go + Cobra + Charm)

If the question is *"what would a 2026-quality, decade-durable M-tooling
CLI look like, ignoring migration cost?"* then the answer is:

- **Go + Cobra** for the CLI shell (the schema, the dispatch, the help,
  the completion, the plugin model).
- **Lip Gloss + Bubble Tea + Glamour** for rendering.
- **Python engines kept as subprocess workers**, communicated with via
  JSON-RPC over stdin/stdout (the same model kubectl uses for `kubectl
  exec`, that Wrangler uses for its dev server, that gh uses for plugin
  extensions).
- **Or rewrite the engines in Go** — feasible given m-cli's bounded
  surface (the lint and fmt rule sets are large but rule-bound; the
  tree-sitter integration is well-supported).

This delivers:

- Single static binary on every OS.
- Sub-10ms startup (vs today's ~150ms).
- The aesthetic of Stripe / gh / fly out of the box.
- Cobra's command-tree as the schema; export to JSON for
  capabilities/completion/IDE consumers.
- A plugin ecosystem on par with Heroku's.

It costs ~3-4 months of focused rewrite. For a hobbyist project, that's
a significant ask. For a project that wants to *be the CLI for M
language tooling*, it's the right investment.

### 7.3 The "Oclif option" — Finalist B, if schema-first is sacred

If the schema-first criterion is the absolute non-negotiable — i.e.
"every fact about the CLI surface must live in a single JSON
manifest, automatically, no glue code" — then **Oclif is the
unambiguous winner**.

`oclif.manifest.json` is auto-generated by `oclif manifest` from
your TypeScript command classes. It is exhaustively complete. Plugins
ship their own manifest fragments which merge at install time. The
audits' three-way drift problem is structurally impossible in Oclif.

The cost: Node startup overhead (~150-250ms unless Bun is adopted)
and the cultural friction of "an M-language CLI written in TypeScript."

## 8. Proposed architecture for m-cli v2

Picking up Finalist A (Go + Cobra + Charm) as the aspirational design.
The architecture decouples the **shell** (grammar, aesthetics, schema)
from the **engine** (M analysis, tree-sitter, lint/fmt rules).

### 8.1 Layered architecture

```
┌──────────────────────────────────────────────────────────┐
│  m (Go binary, ~5 MB static)                             │
│                                                          │
│  ┌─────────────────────────────────────────────────────┐ │
│  │  Cobra dispatcher                                   │ │
│  │  - command tree (built from schema/*.yaml)          │ │
│  │  - flag parsing, validation, help rendering         │ │
│  │  - exit-code policy enforcement                     │ │
│  └─────────────────────────────────────────────────────┘ │
│                                                          │
│  ┌─────────────────────────────────────────────────────┐ │
│  │  Aesthetic layer (Lip Gloss / Glamour / Bubbles)    │ │
│  │  - colours, tables, progress, markdown, diff        │ │
│  │  - TTY-awareness; switches to plain when piped      │ │
│  └─────────────────────────────────────────────────────┘ │
│                                                          │
│  ┌─────────────────────────────────────────────────────┐ │
│  │  Output / schema layer                              │ │
│  │  - one renderer per (verb × format)                 │ │
│  │  - JSON outputs validated against schemas at test   │ │
│  │  - text outputs styled via Lip Gloss                │ │
│  └─────────────────────────────────────────────────────┘ │
│                                                          │
│  ┌─────────────────────────────────────────────────────┐ │
│  │  Worker bridge (JSON-RPC over stdio OR native Go)   │ │
│  │  - lint/fmt/test/coverage drive Python workers (v0) │ │
│  │  - rewrite to native Go (v1+) as bandwidth allows   │ │
│  └─────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────┘
                            ▲
                            │ JSON-RPC
                            ▼
┌──────────────────────────────────────────────────────────┐
│  Python workers (today's m-cli internals, intact)        │
│  - tree-sitter-m parser                                  │
│  - lint rule engine (M-MOD-*, M-XINDX-*, vista, sac)     │
│  - fmt rule engine                                       │
│  - test runner (YDB subprocess)                          │
│  - coverage runner (ZBREAK instrumentation)              │
└──────────────────────────────────────────────────────────┘
```

The Python workers are exactly today's `m_cli.lint.runner`,
`m_cli.fmt.formatter`, etc. They get a thin JSON-RPC wrapper that
exposes them as long-lived subprocesses. The Go shell speaks to them
the same way an LSP client speaks to its server.

### 8.2 Why the worker split is the right initial cut

- **Salvages every dollar of existing engine investment.** Lint rules,
  fmt rules, tree-sitter integration, M-standard loader — all stay.
- **Decouples the rewrite.** Shell ships v1.0; engines stay v0.x. Each
  side iterates independently.
- **Mirrors how the industry's tier-1 CLIs work.** kubectl talks to
  kubelet over gRPC; Wrangler talks to Cloudflare's edge over HTTP;
  Stripe CLI talks to Stripe's API. The shell doesn't *do* anything
  itself — it composes results from workers.
- **Lets us rewrite engines piecemeal** when and if Go-native
  performance/distribution becomes worth it.

### 8.3 The manifest in v2

Cobra's command tree IS the schema, with one extension: a per-command
YAML sidecar declares output schemas, exit codes, examples. The Cobra
build step (`go generate`) reads the YAML and emits Go code, OR
embeds the YAML for runtime access (the latter is simpler and what I'd
recommend).

```yaml
# cmd/engine/status.yaml (embedded via go:embed)
purpose: Print container/image/daemon state
verb_category: status
options:
  - name: --json
    type: bool
    help: Emit JSON
  - name: --quiet
    short: -q
    type: bool
output:
  formats: [text, json]
  json_schema: schemas/engine-status.schema.json
exit_codes:
  0: container running
  1: container not running
examples:
  - command: m engine status
  - command: m engine status --json | jq .container_running
```

The Cobra command's `Run` function reads this and:

1. Builds `--help` text.
2. Validates output against the schema in tests.
3. Surfaces examples in `--help` and in autocompletion specs.

### 8.4 Style guide enforcement

Lip Gloss centralises the colour and glyph vocabulary in one file:

```go
// internal/style/style.go
package style

import "github.com/charmbracelet/lipgloss"

var (
    OK      = lipgloss.NewStyle().Foreground(lipgloss.Color("10")).SetString("✓")
    Fail    = lipgloss.NewStyle().Foreground(lipgloss.Color("9")).SetString("✗")
    Warn    = lipgloss.NewStyle().Foreground(lipgloss.Color("11")).SetString("⚠")
    Unknown = lipgloss.NewStyle().Faint(true).SetString("-")

    Header  = lipgloss.NewStyle().Bold(true)
    Dim     = lipgloss.NewStyle().Faint(true)
    Code    = lipgloss.NewStyle().Foreground(lipgloss.Color("14"))
    Error   = lipgloss.NewStyle().Foreground(lipgloss.Color("9")).Bold(true)
)
```

Every command imports `internal/style`. There is **literally no other
way** to put colour on the screen. The audits' "two severity
vocabularies" / "three missing-value glyphs" problems become
impossible by construction.

### 8.5 Output policy as a framework feature

A `Renderer` interface in `internal/render`:

```go
type Renderer interface {
    RenderText(w io.Writer, data any) error
    RenderJSON(w io.Writer, data any) error
}
```

Each verb returns a typed result. The framework — not the verb —
decides whether to call `RenderText` (TTY, default) or `RenderJSON`
(`--json`, or stdout is not a TTY). Summary lines go to stderr
automatically; bodies go to stdout automatically. `-q` suppresses
summary. **The verb cannot get this wrong.**

## 9. Migration plan — 90-day arc

Assuming Finalist A (Go + Cobra + Charm) is adopted. The plan is
*reversible* at every milestone — if any phase shows it isn't
working, fall back to Finalist D (Typer + Rich) without regret.

### Days 1-14 — Scaffolding & 1 verb proof

- Create `m-cli-v2` repo (or v2 branch).
- Scaffold Cobra project with one verb: `m doctor`.
- Style guide written (`internal/style/`).
- Output renderer framework written.
- Embed YAML for per-verb metadata.
- Export `m capabilities --json` from cobra tree.
- Single static binary builds for Linux/macOS/Windows.
- README + screenshots showcasing the aesthetic.

Goal: prove the architecture on one verb. Decision point: continue
or fall back to Finalist D.

### Days 15-45 — Source-level commands (no engine workers)

The pure-source commands have no dependency on Python:

- `m fmt` (needs tree-sitter-m — Go bindings; or call Python worker).
- `m lint` (same).
- `m capabilities` (pure metadata).
- `m doctor` (already done).
- `m new` (file-system scaffolding; trivial in Go).
- `m ci init` (trivial).
- `m plugins` (`$PATH` discovery).
- `m run` (subprocess YDB).

For `m fmt` and `m lint`, the choice point:
- **Option 1**: Use Go's tree-sitter bindings + reimplement the rules
  natively. Faster, single-binary, but rewrites the lint/fmt rule
  engines.
- **Option 2**: Call the Python worker via JSON-RPC. Slower (subprocess
  startup), but the rule engines stay Python.

I'd recommend Option 2 for v1.0, Option 1 for v2.0+.

### Days 46-70 — Engine-bound commands via workers

- `m test`, `m watch`, `m coverage` — these call into the YDB engine
  via the existing Python runner. Wrap that runner as a JSON-RPC
  worker. Go shell drives it.
- `m engine` and all its actions — the engine driver is small enough
  that rewriting in Go is realistic. Charm's progress bars and
  spinners make the lifecycle verbs *much* prettier.
- `m stdlib *` — reads JSON manifest from m-stdlib. Native Go
  implementation; faster than today.
- `m lsp` — kept as Python; Go shell exec's into it.

### Days 71-85 — Polish & parity

- Every verb gets:
  - Inline examples in `--help`.
  - Markdown rendering via Glamour.
  - JSON output schema-validated by tests.
  - Bash/zsh/fish/powershell completion generated.
- Onboarding flow: first-run detection, friendly intro.
- Error messages with fix hints (compiler-diagnostic style).
- VHS recordings for each major verb (auto-generated screenshots).

### Days 86-90 — Cutover

- Release `m-cli-v2 v1.0`.
- Old `m-cli` deprecated. Compatibility layer: a tiny Python shim that
  shells out to the Go binary for any verb. Existing scripts keep
  working.
- Documentation updated.
- The five audit/plan/architecture docs are archived as historical.

## 10. Open questions

These are the choices that need to be made before day 1.

### Q1 — Realistic or aspirational?

**Default**: Realistic (Finalist D — Typer + Rich + remediation
plan). Easier; preserves all engine investment; ships consistency
faster.

**Alternative**: Aspirational (Finalist A — Go + Cobra + Charm). Best
2026-quality outcome; bigger commitment.

Decision: **incremental Python upgrade or de novo Go rewrite?**

### Q2 — If Go: rewrite engines or keep as workers?

**Default**: Workers in v1.0; native Go rewrite in v2.0+ if and when
worth it.

**Alternative**: Native Go from day 1. Cleaner architecture; longer
delivery.

Decision: **workers first or rewrite first?**

### Q3 — If Python: how much Typer vs how much custom registry?

**Default**: Typer for all verb parsing; small custom registry on top
for output-format policy + exit-code enforcement. Rich for all
rendering.

**Alternative**: Click directly + custom registry. Slightly more
mature than Typer; less ergonomic.

Decision: **Typer (modern, type-hint-driven) or Click (mature,
decorator-driven)?**

### Q4 — Schema authoring format

**Default**: YAML per-command sidecar (`cmd/engine/status.yaml`),
validated against a JSON Schema in `schema/manifest.schema.json`.

**Alternative A**: JSON sidecars. Less human-friendly; more
machine-friendly.

**Alternative B**: Inline in Go struct tags or Python decorators.
No separate file; less external-tooling-friendly.

Decision: **YAML, JSON, or inline?**

### Q5 — Aesthetic: how aggressive?

**Default**: "Stripe / gh aesthetic" — colour discipline, glyphs for
status, markdown in `--help`, progress bars for >500ms operations.
**No animated logos. No bell. No emoji-as-status.**

**Alternative — fancier**: Full Bubble Tea TUI mode for
`m engine status` (live container health), `m watch` (live test
results). More wow; more code.

**Alternative — plainer**: ASCII-only by default; colour only with
explicit `--color` flag.

Decision: **modern-but-restrained, full-TUI, or plain-ASCII?**

### Q6 — Plugin model

**Default**: `$PATH` discovery (git/kubectl style). `m-foo` becomes
`m foo`. Plugin authors don't even need Go/Python familiarity — they
just ship a binary.

**Alternative**: Oclif-style first-class plugin packages with manifest
fragments. Requires plugin authors to use the same framework.

Decision: **PATH-based, framework-based, or both?**

### Q7 — Distribution channel

**Default**: GitHub releases (statically-linked binaries),
`brew install m`, `scoop install m`, `apt install m`. Plus a one-liner
`curl https://m.dev/install.sh | sh`.

**Alternative**: Restricted to GitHub releases. Less reach; less
maintenance.

Decision: **multi-channel from day 1, or release-only initially?**
