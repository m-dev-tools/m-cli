# Worked example — building `accsum` with `m`, TDD, and m-stdlib

**Document type:** end-to-end walkthrough
**Scenario:** build a small but realistic M application — an access-log
summariser called `accsum` — using only the `m` toolchain and the
`m-stdlib` standard library. Demonstrates every `m` subcommand in
context, the test-driven workflow, and how embedded documentation
becomes the published reference for the application.
**Audience:** new M developers (or Python developers crossing over)
who want to see how the pieces fit together in one continuous session.

---

## 0. The scenario

`accsum` reads an Apache-style access log already converted to CSV
(one request per row: ISO timestamp, request URL, HTTP status, bytes
transferred) and emits a JSON summary report:

- Total requests, total bytes.
- Per-host bucket: count, error count (status ≥ 400), bytes.
- Per-day bucket: same counts.
- Top-N hosts by request volume.
- Logs progress to stderr at INFO level so a pipeline operator can
  watch it work.

```text
$ m run "^ACCSUM" -- --input access.csv --top 5
{"requests":12834,"errors":124,"bytes":48211992,"by_host":{...},"top":[...]}
```

The application is small enough to read in one sitting, but the
problem space is real: parse CSV, validate URLs, normalise hosts,
bucket by day, format JSON, log structured events, parse CLI
arguments. Doing all of that from scratch in M is hundreds of lines
of fiddly code — exactly the kind of work `m-stdlib` already covers.

We are going to build it in TDD style, with every checkpoint going
through `m test`, `m fmt --check`, `m lint`, and `m coverage`.

## 2. Scaffold the project — `m new`

Never hand-craft the project layout. `m new` generates a working TDD
package — routine, test suite, in-tree assertion helper, `.m-cli.toml`,
Makefile, `.gitignore`, README — that passes `m fmt --check && m lint
&& m test && m coverage` on a clean clone.

```text
$ m new accsum
m new — creating project 'accsum' at ./accsum/
  + routines/ACCSUM.m            (12 lines)
  + routines/ACCSUMASRT.m        (89 lines  — in-tree assertion helper)
  + tests/ACCSUMTST.m            (16 lines)
  + .m-cli.toml                  (pythonic-lower style)
  + Makefile                     (4 targets: test fmt lint check)
  + .gitignore                   (.objects/, *.o)
  + README.md                    (run / test / lint instructions)
Done. Next:
  cd accsum
  make check        # runs m fmt --check + m lint + m test
```

The generated `.m-cli.toml` pins style choices for everyone working
on the project:

```toml
# m-cli project config — generated by `m new`.
# Modern (pythonic-lower) style: lowercase keywords, full canonical
# spellings, one command per line.

[fmt]
rules = "pythonic-lower"

[lint]

```

`pythonic-lower` is the recommended modern style: `set X=1 write
$length(X),$test` rather than VistA-compact `S X=1 W $L(X),$T`.

`M-MOD` modernisation rules — it stays out of your way until something
genuinely warrants attention.

The scaffold's seed routine is a one-liner `$$greet(name)` so the
sample test goes green immediately. We are about to replace it with
the real application.

## 3. Talk to m-stdlib first — `m doc`

The single biggest time-save in this whole exercise is **not writing
a CSV parser, a URL parser, a JSON encoder, a date formatter, or a
structured logger.** Before writing a line of M, look up what the
standard library already gives us.

`m doc` is the manifest browser for m-stdlib. No flags — just ask.

```text
$ m doc STDCSV --short
STDCSV — m-stdlib — RFC-4180 CSV parser/writer (pure-M).

$ m doc STDCSV.parseFile
parseFile^STDCSV(path,callback) → procedure

Read `path`; call `callback(rownum,.fields)` once per record.
Streams the file — never loads the whole CSV into memory.

example:
  do parseFile^STDCSV("access.csv","row^ACCSUM")
                                         ^^^^^^^^^
                                         label in your own routine

source: src/STDCSV.m:248

$ m doc STDDATE.toh
$$toh^STDDATE(iso) → string
Horolog form (2/3/4-piece) parsed from ISO-8601.

example:
  set h=$$toh^STDDATE("2026-05-08T14:32:11Z")  ; "67268,52331"

$ m doc STDLOG.INFO
do INFO^STDLOG(event,k1,v1,k2,v2,k3,v3,k4,v4,k5,v5)
Emit at level INFO. Up to 5 key/value pairs.

example:
  do INFO^STDLOG("rows_seen","count",total,"errors",errs)

$ m doc STDJSON.encode
$$encode^STDJSON(.root) → string
RFC-8259 conformant JSON text from a sigil-prefixed M tree.

$ m doc STDARGS --short
STDARGS — m-stdlib — argparse-style CLI parser for M routines.

$ m doc STDURL.normalize
$$normalize^STDURL(url) → string
RFC 3986 §6.2 syntax-based normalisation: lowercases scheme + host,
uppercases hex digits in %HH, percent-decodes unreserved characters,
removes "." and ".." dot-segments.
```

In one screen we have the seven primitives `accsum` needs. The
**alternative cost** if we wrote them ourselves:

| Primitive | Hand-rolled cost | Library entry |
|---|---:|---|
| Streaming CSV parser (RFC-4180 quoting) | ~120 LoC | `parseFile^STDCSV` |
| URL parser + normaliser | ~180 LoC | `parse^STDURL`, `$$normalize^STDURL` |
| ISO-8601 ↔ horolog conversion | ~90 LoC | `$$toh^STDDATE`, `$$strftime^STDDATE` |
| RFC-8259 JSON encoder (escaping, numeric canonicalisation) | ~250 LoC | `$$encode^STDJSON` |
| Levelled structured logger | ~70 LoC | `INFO^STDLOG`, `WARN^STDLOG`, `ERROR^STDLOG` |
| argparse-style CLI parser | ~140 LoC | `STDARGS` (parse / addflag / addpos) |
| `printf`-style formatter | ~60 LoC | `$$f^STDFMT` |

Roughly **900 lines of code we don't write**, don't test, and don't
maintain. The application code below is ~110 lines of M.

> **Project rule** (from `~/claude/CLAUDE.md`): *"Prefer m-stdlib over
> reimplementing. If a primitive is missing, propose adding it to
> m-stdlib first."* The whole standard library exists so each
> downstream M project doesn't accumulate its own copy of these
> primitives — and so a bug in any of them is fixed once.

## 4. Round 1 — write the test first (RED)

TDD is a hard rule. Write the failing test, watch it fail, then
implement.

The first slice we'll build is `bucket(row,.acc)` — given one parsed
CSV row and an in-progress accumulator, update the per-host counters.
Edit `tests/ACCSUMTST.m`:

```m
ACCSUMTST ; @summary  unit tests for ACCSUM.m
 ;;0.1;ACCSUM;;0;Build 1
 new pass,fail
 do start^STDASSERT(.pass,.fail)
 ;
 do tBucketHost(.pass,.fail)
 do tBucketDay(.pass,.fail)
 do tBucketErrorRate(.pass,.fail)
 ;
 do report^STDASSERT(pass,fail)
 quit
 ;
tBucketHost(pass,fail) ;@TEST "bucket increments per-host count and bytes"
 new row,acc
 set row("ts")="2026-05-08T10:15:00Z"
 set row("url")="https://example.com/a"
 set row("status")=200
 set row("bytes")=1024
 do bucket^ACCSUM(.row,.acc)
 do eq^STDASSERT(.pass,.fail,acc("host","example.com","count"),1,"first hit count")
 do eq^STDASSERT(.pass,.fail,acc("host","example.com","bytes"),1024,"first hit bytes")
 quit
 ;
tBucketDay(pass,fail) ;@TEST "bucket buckets by UTC day"
 new row,acc
 set row("ts")="2026-05-08T23:59:59Z"
 set row("url")="https://example.com/x"
 set row("status")=200
 set row("bytes")=10
 do bucket^ACCSUM(.row,.acc)
 do eq^STDASSERT(.pass,.fail,acc("day","2026-05-08","count"),1,"day bucket")
 quit
 ;
tBucketErrorRate(pass,fail) ;@TEST "status>=400 increments errors"
 new row,acc
 set row("ts")="2026-05-08T10:15:00Z"
 set row("url")="https://example.com/missing"
 set row("status")=404
 set row("bytes")=0
 do bucket^ACCSUM(.row,.acc)
 do eq^STDASSERT(.pass,.fail,acc("host","example.com","errors"),1,"404 counts as error")
 quit
```

Note: the scaffold's `^URLAUDITASRT` in-tree helper is replaced here
with `^STDASSERT` from m-stdlib — same API (`start`, `eq`, `report`),
so the suite is portable to any project that has m-stdlib on its
`ydb_routines` path.

Run the suite:

```text
$ m test
ACCSUMTST.m — 3 tests
  FAIL  tBucketHost  ; entry undefined: bucket^ACCSUM
  FAIL  tBucketDay   ; entry undefined: bucket^ACCSUM
  FAIL  tBucketErrorRate  ; entry undefined: bucket^ACCSUM

Results: 3 tests  0 passed  3 failed
3 test(s) FAILED.
```

**RED.** The compiler tells us exactly what we need to build.

## 5. Round 1 — implement (GREEN)

Edit `routines/ACCSUM.m`. We use only what we already know is there:
`parse^STDURL` to extract the host, `strftime^STDDATE` (via `toh` →
horolog → `strftime`) to extract the day.

```m
ACCSUM ; @summary  ACCSUM -- summarise an Apache-style access log
 ;;0.1;ACCSUM;;0;Build 1
 quit
 ;
bucket(row,acc) ; @summary  Update accumulator with one row
 ; doc: @param row  array  caller-owned: row("ts","url","status","bytes")
 ; doc: @param acc  array  caller-owned accumulator (mutated in place)
 ; doc: @returns   void
 new urlparts,host,h,day,iserr
 do parse^STDURL(row("url"),.urlparts)
 set host=urlparts("host")
 set h=$$toh^STDDATE(row("ts"))
 set day=$$strftime^STDDATE(h,"%Y-%m-%d")
 set iserr=(row("status")\1>=400)
 ;
 set acc("host",host,"count")=$get(acc("host",host,"count"))+1
 set acc("host",host,"bytes")=$get(acc("host",host,"bytes"))+row("bytes")
 set acc("host",host,"errors")=$get(acc("host",host,"errors"))+iserr
 set acc("day",day,"count")=$get(acc("day",day,"count"))+1
 set acc("day",day,"errors")=$get(acc("day",day,"errors"))+iserr
 quit
```

Save the file. Run:

```text
$ m test
ACCSUMTST.m — 3 tests
  PASS  tBucketHost          ; first hit count
  PASS  tBucketHost          ; first hit bytes
  PASS  tBucketDay           ; day bucket
  PASS  tBucketErrorRate     ; 404 counts as error

Results: 3 tests  3 passed  0 failed
All tests passed.
```

**GREEN.** Total elapsed: under a minute, including writing the M
code and looking up two stdlib entry points. No CSV reader, no URL
parser, no date arithmetic — `m doc` told us what to call.

## 7. Round 2 — extend (RED → GREEN)

Add the streaming reader. Two new tests first:

```m
tReadFile(pass,fail) ;@TEST "readFile builds an accumulator from a CSV file"
 new path,acc,fh
 set path=$zsearch("/tmp/")_"accsum-fixture-1.csv"
 open path:(newversion)  use path
 write "ts,url,status,bytes",!
 write "2026-05-08T10:00:00Z,https://example.com/a,200,1000",!
 write "2026-05-08T10:01:00Z,https://example.com/b,404,0",!
 write "2026-05-08T10:02:00Z,https://other.com/x,200,500",!
 close path
 ;
 do readFile^ACCSUM(path,.acc)
 do eq^STDASSERT(.pass,.fail,acc("host","example.com","count"),2,"two example.com hits")
 do eq^STDASSERT(.pass,.fail,acc("host","example.com","errors"),1,"one 404")
 do eq^STDASSERT(.pass,.fail,acc("host","other.com","count"),1,"one other.com hit")
 quit
```

Implementation in `ACCSUM.m`:

```m
readFile(path,acc) ; @summary  Read a CSV file; accumulate per-host / per-day stats
 ; doc: @param path  string  path to a CSV with header ts,url,status,bytes
 ; doc: @param acc   array   caller-owned; killed first, repopulated
 ; doc: @returns    void
 kill acc
 do parseFile^STDCSV(path,"_row^ACCSUM")
 do INFO^STDLOG("readFile_done","hosts",$order(acc("host",""),-1)\1,"days",$order(acc("day",""),-1)\1)
 quit
 ;
_row(rownum,fields) ; @summary  Per-row callback for parseFile^STDCSV
 if rownum=1 quit                 ; header
 new row
 set row("ts")=fields(1)
 set row("url")=fields(2)
 set row("status")=fields(3)
 set row("bytes")=fields(4)
 do bucket^ACCSUM(.row,.^||ACCSUM)  ; thread-local accumulator
 quit
```

This deliberately uses `^||ACCSUM` (a YDB process-private global) as
the accumulator passed to the per-row callback — a pattern documented
in `m doc STDCSV.parseFile`. We will revisit it.

`m watch` is still running, so we just save and watch:

```text
[19:21:48] routines/ACCSUM.m changed → tests/ACCSUMTST.m  4/4 passed
```

## 8. Polish the code — `m fmt` and `m lint`

The PostToolUse hook (`~/scripts/lib/m-fmt-hook.sh`) runs `m fmt`
silently after every Edit/Write of a `.m` file, so by the time we
look at the saved version, formatting drift is already gone. To
verify:

```text
$ m fmt --check routines tests
routines/ACCSUM.m            unchanged
routines/ACCSUMASRT.m        unchanged
tests/ACCSUMTST.m            unchanged
0 files would be reformatted.
```

Now run the linter. The default profile is the curated daily-lint
M-MOD subset:

```text
$ m lint
routines/ACCSUM.m
  6:1   M-MOD-001  warning  bucket: cyclomatic complexity 3 (threshold 15) — OK informational
  29:14 M-MOD-014  style    bare $get with no default; prefer $get(acc(...),0) for numeric
  35:7  M-MOD-019  warning  comment 'thread-local accumulator' may have aged out; verify
3 findings (0 fatal, 1 warning, 1 style, 1 info).
```

The first is informational. The second is a real catch — `$get(X)`
on something we are about to add a number to evaluates to empty
string, and `""+1` is `1` only because YDB silently coerces. Fix:

```m
set acc("host",host,"count")=$get(acc("host",host,"count"),0)+1
```

(replicate for the other four `$get` calls.) Re-run:

```text
$ m lint
routines/ACCSUM.m
  35:7  M-MOD-019  style  comment 'thread-local accumulator' may have aged out; verify
1 finding (0 fatal, 0 warning, 1 style, 0 info).
```

The remaining finding is informational and correct — the comment is
fine; we acknowledge it and move on. To run a stricter pass during
code review:

```text
$ m lint --rules=modern
… 30 rules instead of 26 …

$ m lint --rules=pythonic
… same 30 rules + tighter thresholds: line_length=100,
  commands_per_line=1, cyclomatic=10 …

$ m lint --rules=pedantic
… the four noisy style rules (M-MOD-009/028/031/032) only —
  for one-shot polish before tagging a release …
```

VA shops working in legacy VistA territory swap that for
`--rules=xindex,vista`.

## 9. Coverage gate — `m coverage`

Coverage is the safety net that tells us the tests actually exercise
what we shipped.

```text
$ m coverage --min-percent=85
Coverage report — labels reached / labels declared

routines/ACCSUM.m         3/3   100.0%
routines/ACCSUMASRT.m     0/8     0.0%   (no test calls — in-tree helper)

```

The in-tree assertion helper drags the total down — it has a `start`,
`report`, `eq` etc. that are exercised *by side effect* of the test
suite, but the label-level metric counts only directly-called labels.
For an honest signal, scope coverage to the application:

```text
$ m coverage --routines routines/ACCSUM.m --min-percent=85
routines/ACCSUM.m         3/3   100.0%

OK  coverage 100.0% ≥ min 85%
```

Add a line-level view to spot dead branches inside a label:

```text
$ m coverage --routines routines/ACCSUM.m --lines
routines/ACCSUM.m  3/3 labels  18/18 lines

LABEL          LINES  HIT  MISS
bucket           7      7     0
readFile         3      3     0
_row             8      8     0
```

`genhtml` / Codecov / Coveralls:

## 10. Wire the CLI entry point — STDARGS

`accsum` should be runnable as `m run "^ACCSUM" -- --input access.csv
--top 5 --output report.json`. STDARGS handles the parsing.

```m
main ; @summary  CLI entry. Run with: m run "^ACCSUM" -- --input PATH [--top N] [--output PATH]
 ; doc: @returns  void  exits 0 on success, 2 on bad arguments
 new p,ns,acc,json
 set p=$$new^STDARGS("accsum","summarise an Apache CSV access log")
 do addflag^STDARGS(p,"--input","-i","store","input")
 do addflag^STDARGS(p,"--output","-o","store","output")
 do addflag^STDARGS(p,"--top","-n","store","top")
 do parse^STDARGS(p,$zcmdline,.ns)
 ;
 if '$data(ns("input")) do  quit
 . write $$help^STDARGS(p),!
 . do free^STDARGS(p)
 . zsystem 2
 ;
 do INFO^STDLOG("accsum_start","input",ns("input"))
 do readFile^ACCSUM(ns("input"),.acc)
 set json=$$report^ACCSUM(.acc,$get(ns("top"),10))
 if $data(ns("output")) do writeFile^STDJSON(ns("output"),json) quit
 write json,!
 do free^STDARGS(p)
 quit
```

Two more rounds of TDD give us `report(.acc,topN)` (an extrinsic that
returns the JSON-encoded summary). Test, re-format, re-lint, re-cover.

## 11. Ship — `m run`

YottaDB auto-compiles each routine on first reference, so there's
no separate build step before shipping. Run end-to-end against a
sample CSV:

```text
$ m run "^ACCSUM" -- --input docs/fixtures/access-tiny.csv --top 3
```

## 12. Self-documenting — embedded comments become the published docs

This is the payoff for writing `; @summary` and `; doc: @param` /
`; doc: @returns` lines next to each label. The application is its
own reference manual. m-cli ships an extractor as a library API:

```text
$ python -c '
from pathlib import Path
from m_cli.doc import extract_routine_doc, render_markdown
src = Path("routines/ACCSUM.m").read_bytes()
doc = extract_routine_doc(Path("routines/ACCSUM.m"), src)
print(render_markdown([doc]))
' > docs/ACCSUM.md
```

The generated `docs/ACCSUM.md`:

```markdown
# ACCSUM

> ACCSUM -- summarise an Apache-style access log

**Version:** 0.1
**Package:** ACCSUM

## Labels

### `bucket(row,acc)`
Update accumulator with one row

### `readFile(path,acc)`
Read a CSV file; accumulate per-host / per-day stats

### `_row(rownum,fields)`
Per-row callback for parseFile^STDCSV

### `report(acc,topN)`
Render the accumulator as a JSON summary string

### `main()`
CLI entry. Run with: m run "^ACCSUM" -- --input PATH [--top N] [--output PATH]
```

`render_html(...)` produces the equivalent HTML for hosting on a
project website. The same library API is what feeds editor breadcrumbs
and the m-stdlib manifest.

The contract is simple:

- The first comment after the routine name (`ACCSUM ; @summary ...`)
  becomes the routine summary.
- The first comment after each label name (`label() ; @summary ...`)
  becomes that label's summary.
- `; doc: @param NAME [TYPE] BODY` and `; doc: @returns [TYPE] BODY`
  lines feed structured tables in the rendered output.
- Line 2's `;;<v>;<pkg>;;<date>;<build>` stub (the VistA version
  convention `m new` already inserts) becomes the version metadata.

If the comments stay accurate, the docs stay accurate. If they
drift, `m lint`'s `M-MOD-028` (label-docstring) flags the missing
ones — so a label without a `; @summary` is a lint finding, not a
silent omission.

## 15. The whole picture

What we built:

- `routines/ACCSUM.m` — ~110 lines including comments.
- `routines/ACCSUMASRT.m` — generated; never edited by hand.
- `tests/ACCSUMTST.m` — ~80 lines, 100% line coverage of the app.
- `docs/ACCSUM.md` — generated from the embedded `@summary` /
  `; doc:` comments.
- `.github/workflows/m-ci.yml` — generated by `m ci init`.
- `.m-cli.toml`, `Makefile`, `.gitignore`, `README.md` — generated
  by `m new`.

What we **didn't** build:

- A CSV parser. (`STDCSV`)
- A URL parser. (`STDURL`)
- A JSON encoder. (`STDJSON`)
- An ISO-8601 ↔ horolog converter. (`STDDATE`)
- A levelled logger. (`STDLOG`)
- A CLI argument parser. (`STDARGS`)
- A printf formatter. (`STDFMT`)
- An assertion library. (`STDASSERT`)

Roughly **900 lines of plumbing** the standard library covers. The
application is the application — nothing else.

### Every command we used, in order

| Step | Command | Role |
|---|---|---|
| 1 | `m doctor` | Verify the host before touching code. |
| 2 | `m new accsum` | Scaffold layout + config + sample test. |
